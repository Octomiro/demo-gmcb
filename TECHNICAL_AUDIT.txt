3. AI / PROCESSING PIPELINE
============================

------------------------------------------------------------
Primary model    | yolo26m_BB_barcode_date.pt (YOLOv8/11m, 131 MB)
Secondary model  | yolo26-BB(date).pt (date-only, 44 MB) — runs in parallel via ThreadPoolExecutor(max_workers=1)
Engine           | PyTorch 2.10.0 + CUDA 12.6 on NVIDIA RTX 4090
Inference input  | imgsz=640, conf=0.45, half=False (FP32)
Tracker          | ByteTrack (Ultralytics built-in, persisted via temp YAML)

Pipeline 2: Anomaly Detection (pipeline_anomaly)
-------------------------------------------------
YOLO model       | yolo26m_seg_farine_FV.pt (segmentation, 54 MB)
EfficientAD      | Teacher (32MB) + Student (46MB) + Autoencoder (4.4MB) — all on CUDA
AD input         | 256x256 letterboxed crops, batched inference
AD strategy      | MAJORITY vote across up to 7 scans per packet
AD threshold     | 5000.0
Frame skip       | ANOMALY_FRAME_SKIP = 2
Scanning zone    | Entry at 80% frame width, exit at 20%

Both pipelines run on GPU (CUDA). DEVICE = 'cuda'.

Inference time estimate:
- YOLO inference on RTX 4090 at 640px FP32: ~8-15ms per frame
- EfficientAD batched (teacher+student+autoencoder): ~5-15ms per batch
- Total detector loop (YOLO + tracking + overlay build): ~20-40ms → ~25-50 det_fps

Buffering inventory:
--------------------
Location            | Type                              | Size
--------------------|-----------------------------------|-----------------------
V4L2 driver         | Kernel ring buffer                | CAP_PROP_BUFFERSIZE=2
_raw_frame          | Single latest frame (overwritten) | 1 frame
_raw_history        | Deque for compositor sync         | 24 frames max
_det_frame          | Single frame for detector (copy)  | 1 frame
_jpeg_bytes         | Pre-encoded JPEG (single latest)  | 1 frame
Nginx               | DISABLED (proxy_buffering off)    | 0
Browser <img>       | Depends on browser impl           | ~1 frame

Where latency is introduced:
-----------------------------
1. V4L2 double-buffer: ~16ms (1 frame at 60fps)
2. Frame skip: Every 2nd frame → detector sees 30fps of the 60fps input
3. YOLO inference: ~10-20ms per detection frame
4. Compositor: Overlay drawn on LATEST raw frame (not the detection frame)
   — eliminates YOLO latency from display but causes ~1 frame bbox drift
5. JPEG encoding: TurboJPEG, ~1-3ms
6. MJPEG streaming: Generator yields at target fps (default 25fps sleep interval)
7. Nginx proxy: Minimal (buffering disabled)
8. Browser <img> MJPEG: ~1 frame buffered by browser

Frames are NOT queued for detection. The detector always picks the latest
available frame and skips intermediate ones. The overlay from the last
detection is drawn on the current raw frame → near-zero display latency.

6. CPU vs GPU USAGE
====================

Component                              | Runs On   | Notes
---------------------------------------|-----------|------------------------------
Video capture (V4L2 MJPEG decode)      | CPU       | OpenCV decodes MJPEG → BGR
Frame rotation                         | CPU       | cv2.rotate() in reader thread
YOLO inference (both pipelines)        | GPU       | FP32, RTX 4090, CUDA
ByteTrack                              | CPU       | Lightweight tracking logic
Secondary date model                   | GPU       | Parallel ThreadPoolExecutor
EfficientAD (teacher/student/AE)       | GPU       | Batched inference, CUDA
Bbox association / overlay logic       | CPU       | In detector thread
Compositor drawing (rectangle, putText)| CPU       | In compositor thread
JPEG encoding                          | CPU       | TurboJPEG (SIMD, GIL-free)
Flask/gevent MJPEG serving             | CPU       | Greenlets
DB writes                              | CPU       | Async queue via DBWriter

GPU utilization note:
The RTX 4090 is extremely powerful. Running YOLO at FP32 on 640px wastes some
potential — FP16 (half precision) would nearly double throughput — but FP32 is
the correct choice for maximizing detection accuracy on long production shifts.

9. RECOMMENDATIONS (stability + detection accuracy priority)
=============================================================

1. Keep DETECTOR_FRAME_SKIP=2 — Processing every 2nd frame at 60fps gives
   30 detections/sec. Excellent coverage with GPU headroom for thermal
   stability during long shifts.

2. Keep half=False (FP32) — FP32 gives marginally better precision for
   barcode/date classification edge cases. RTX 4090 handles FP32 at 30fps
   easily without thermal pressure.

3. Consider reducing CAMERA_FPS from 60 to 30 — You skip every 2nd frame
   anyway, so the reader discards 50% of captured frames. Capturing at
   30fps directly would:
   - Halve USB bandwidth (less driver contention during long shifts)
   - Reduce reader thread CPU load
   - Same detection throughput
   - Better thermal/stability profile for prolonged operation

4. The 3-thread architecture (reader/detector/compositor decoupled),
   TurboJPEG, gevent WSGI, and the "display latest frame, overlay from
   last detection" approach is a well-designed low-latency pipeline.
   No architectural changes needed.
